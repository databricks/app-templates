Key Findings
Inverted U-curve: More workers != better performance. Medium peaks at 4 workers, Large at 8. Beyond that, process overhead (memory, context switching) hurts throughput. Each uvicorn worker is a full Python process; the async-I/O mock agent doesn't benefit from extra processes.
Large 8w still climbing: At 200 concurrent users, Large 8w showed perfectly linear QPS scaling (192.8 req/s) with flat p50 latency (110ms). The true ceiling requires testing with 400+ users.
Workers > Compute: With 4 workers, Medium and Large perform identically (~70-87 QPS). The advantage of larger compute only appears when paired with enough workers to use the CPUs.
All configs exceed 10 QPS target by an order of magnitude. Even the worst config (Medium 2w) sustained 56 QPS.
Why 4 Workers Beats 8 Workers on Medium

This is counterintuitive but well-explained by how Python multiprocessing works:

  • Each uvicorn worker is a full Python process — not a thread. On a Medium compute instance (4 CPUs), running 8 workers means 2 processes compete for every CPU core.
  • The mock agent is purely async I/O (asyncio.sleep) — it doesn't do CPU-bound work. A single async worker can handle hundreds of concurrent streams because it just awaits timers. Extra processes add zero parallelism benefit for this workload.
  • More processes = more overhead — context switching between 8 processes on 4 cores wastes CPU cycles. Each process also consumes ~50-100MB of RAM for the Python interpreter + FastAPI + MLflow stack, increasing memory pressure.
  • The OS scheduler thrashes — with 8 workers on 4 CPUs, the kernel constantly preempts processes. This shows up directly in the latency: 8w has p50=190ms vs 4w's p50=110ms, meaning each request waits ~80ms longer just due to scheduling overhead.
  • The sweet spot is ~1 worker per CPU — this is confirmed by both compute sizes: Medium optimal is 4 workers on 4 CPUs, Large optimal is 8 workers on 16 CPUs (both roughly 1:2 worker-to-CPU ratio actually, since Large has 16 CPUs).

Note: With a real CPU-bound agent (e.g., doing local inference or heavy data processing), more workers could help. But for typical LLM agents that are I/O-bound (waiting on model API calls), fewer workers with async concurrency is optimal.

Mock Agent

A Python FastAPI app using mlflow.genai.agent_server with @invoke() and @stream() decorators. The mock agent simulates a realistic LLM streaming response without any external API calls, so we measure pure Apps infrastructure throughput.

FrameworkFastAPI + MLflow AgentServer + Uvicorn
Streaming ProtocolServer-Sent Events (SSE) via POST /invocations
Simulated Response1 tool call (get_current_time) + 80 text chunks with 10ms inter-chunk delay (~800ms total stream)
Worker ConfigUvicorn workers set via UVICORN_WORKERS env var, deployed with databricks bundle deploy --var="workers=N"

Request & Response Format

Request (POST /invocations):

{
  "input": [
    {
      "role": "user",
      "content": "What time is it right now, and what time will it be 2 hours and 32 minutes from now?"
    }
  ],
  "stream": true
}

Response (SSE stream — each line is data: {json}):

data: {"type": "response.output_item.added", "item": {"type": "function_call", "name": "get_current_time", ...}}
data: {"type": "response.function_call_arguments.done", ...}
data: {"type": "response.output_item.done", "item": {"type": "function_call", ...}}
data: {"type": "response.output_item.done", "item": {"type": "function_call_output", "output": "2026-04-01T..."}}
data: {"type": "response.output_item.added", "item": {"type": "message", ...}}
data: {"type": "response.output_text.delta", "delta": "The current time is ..."}   ← repeated 80x, 10ms apart
data: {"type": "response.output_text.delta", "delta": "Time zones are regions ..."}
... (78 more text delta chunks)
data: {"type": "response.output_item.done", "item": {"type": "message", ...}}
data: [DONE]

Test Procedure (per worker config)

  1. Deploy mock agent with target worker count: databricks bundle deploy --target {medium|large} --var="workers=N"
  2. Trigger app restart: databricks apps deploy {app-name} --source-code-path ...
  3. Wait for app state = RUNNING and compute state = ACTIVE (up to 6 min)
  4. Run healthcheck: single POST to /invocations with stream=false
  5. Warmup: 5 concurrent users for 30 seconds
  6. Load test: step-ramp from 20→200 users, collecting Locust CSV + HTML reports

Caveats

  • Medium 6w had 10.4% failure rate due to a workspace outage mid-test (“Databricks Apps is disabled for organization”). True clean performance is likely between 4w and 8w.
  • Large 4w sweep was fully corrupted by the same outage. The original full-ramp result (70.0 QPS, 0.1% fails) is used instead.
  • The staging workspace intermittently returned the “disabled for organization” error, which corrupted several runs and required retries.
  • These results measure Apps infrastructure throughput only. Real agents with LLM calls will be bottlenecked by model API latency, not App throughput.
