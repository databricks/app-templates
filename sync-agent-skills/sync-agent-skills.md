# Sync Agent Skills

Assemble the agent-template skills under `.claude/skills/` into a **single** skill,
`databricks-agent-templates`, in the public
[databricks-agent-skills](https://github.com/databricks/databricks-agent-skills)
repo — a router `SKILL.md` plus one `references/<source>/` directory per source
skill. One skill with internal references matches that repo's house style (e.g.
`databricks-pipelines` is one skill with ~40 references), rather than shipping
~18 sibling skills.

This repo is the **source of truth**; the sync is **one-way** (a clean overwrite
of the target skill). It's run manually in Claude Code and opens the downstream
PR via a **browser compare URL** using your own GitHub session — no token, App,
or GitHub Action, and it works even from a fork.

## Configuration

Read `sync-agent-skills/sync-config.yml`:
- `skill`: the target skill `name` + `description` (become the SKILL.md frontmatter).
- `references`: list of `{ source, group, summary, related? }`. `source` is a dir
  under `.claude/skills/`; `group`/`summary` build the router index; `related` is
  an optional cross-link body.
- `placeholders`: literal `{{...}}` → value substitutions applied to every file.

## Task

### 1. Locate repositories and check working trees

**In parallel:**
- **Source (this repo):** the current app-templates working directory.
- **Target (agent-skills):** `~/projects/databricks-agent-skills` or
  `~/databricks-agent-skills`. If not found, tell the user and stop.

Check both for uncommitted changes. If the agent-skills repo is dirty, or this
repo has uncommitted changes under `.claude/skills/`, ask the user — do **not**
stash or discard. Changes outside the sync scope are fine.

### 2. Update to latest

Pull the latest default branch for each repo (`main` for both). If a pull fails,
ask the user.

### 3. Rebuild the skill

Let `TARGET = skills/databricks-agent-templates` in the agent-skills repo.
**Remove `TARGET` entirely** (clean overwrite), then recreate it.

For each `references` entry, build `TARGET/references/<source>/<source>.md` from
`.claude/skills/<source>/SKILL.md`:

1. **Strip the YAML frontmatter** (keep the body).
2. **Render placeholders** (every `placeholders` key → its value).
3. **Rewrite sibling references:** where the body says `` **<other-source>** skill ``
   or `` `<other-source>` skill `` for another `source` in the config, rewrite it
   to an intra-skill link: `` [`<other-source>`](../<other-source>/<other-source>.md) reference ``.
   Leave template-only names that are **not** `source` dirs (`add-tools`,
   `agent-memory`, `modify-agent`) as plain prose.
4. **Inject the cross-link:** if the entry has `related`, insert
   `> **Related skill(s):** <related>` as a blockquote at the **top** of the doc
   (before the first heading), with a blank line after it.
5. **Copy examples:** if `.claude/skills/<source>/examples/` exists, copy it to
   `TARGET/references/<source>/examples/`. (Relative `examples/...` links in the
   doc then resolve unchanged — do not rewrite them.)

### 4. Write the router `SKILL.md`

`TARGET/SKILL.md`:
- Frontmatter: `name: <skill.name>` and `description: "<skill.description>"`.
- A short intro, then this note:
  `> These references mirror the agent templates in [databricks/app-templates](https://github.com/databricks/app-templates) (.claude/skills/). Where a topic overlaps a dedicated skill, the reference defers to it.`
- One `## <group>` section per `group` (in the order groups first appear in the
  config), each listing its references as:
  `- [<source>](references/<source>/<source>.md) — <summary>`

### 5. Provenance marker

Write `TARGET/.synced-from`:

```
Generated by app-templates/sync-agent-skills (sync-agent-skills.md)
Source of truth: github.com/databricks/app-templates (.claude/skills/)
Do not hand-edit — changes go to app-templates.
```

### 6. Assert clean render

Confirm no `{{` placeholder survives anywhere under `TARGET`. If any remain,
stop and report which files.

### 7. Generate and validate the manifest

In the agent-skills repo:

```bash
python3 scripts/skills.py generate
python3 scripts/skills.py validate
```

`generate` synthesizes `agents/openai.yaml` + icons and rebuilds `manifest.json`.
The skill name `databricks-agent-templates` auto-titlecases to a correct display
name, so no `DISPLAY_NAME_OVERRIDES` / `SKILL_METADATA` entry is needed. If
`validate` fails, fix the underlying issue — do not hand-edit `manifest.json`.

### 8. Summarize changes

Show the user a short bulleted summary of what was added / changed / removed.

### 9. Create a branch, commit, and push

In the agent-skills repo:

```bash
git checkout -b update/agent-template-skills-<YYYY-MM-DD>
```

Stage all changes (including deletions) and commit with a single-line message:

```
Update agent-template skills
```

Push to a remote **you can write to**:
- Direct write to databricks-agent-skills → push to `origin`.
- Otherwise push to **your fork** (e.g. `<your-gh-username>/databricks-agent-skills`).

### 10. Open the pull request (browser, no token)

Open the GitHub compare page pre-filled, matching where you pushed:

```bash
# branch on databricks-agent-skills directly:
open "https://github.com/databricks/databricks-agent-skills/compare/main...<branch>?expand=1&title=<url-encoded-title>&body=<url-encoded-body>"

# pushed to your fork:
open "https://github.com/databricks/databricks-agent-skills/compare/main...<your-gh-username>:<branch>?expand=1&title=<url-encoded-title>&body=<url-encoded-body>"
```

A maintainer reviews and merges — this runbook only proposes. Title:
`Update agent-template skills`; body: a short summary + bulleted changes.

**Safety (both repos are public):** never include secrets, tokens, real
workspace IDs/URLs, or PATs in the branch or PR; keep the description about what
the skill *does*.

## Error handling

- A `source` in `sync-config.yml` doesn't exist → stop and help fix the config.
- A new source skill you want mirrored isn't in `references` → it won't be
  included. Add a `{ source, group, summary }` entry (and `related` if it
  overlaps an existing skill) before re-running.
- Never force-push or overwrite uncommitted work in either repo.
