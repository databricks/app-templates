# Sync Agent Skills

Mirror the agent-template skills under `.claude/skills/` into the public
[databricks-agent-skills](https://github.com/databricks/databricks-agent-skills)
repo. This repo is the **source of truth**; the sync is **one-way** and the
downstream copies are generated mirrors, so this is a clean
overwrite — no three-way merge.

This is run manually by a person in Claude Code; it opens the downstream PR via a
**browser compare URL** using your own GitHub session — and works even if you only have a fork of
databricks-agent-skills (not direct write).

## Configuration

Read `sync-agent-skills/sync-config.yml`:
- `skills`: list of `{ source, target, related? }`. `source` is a dir under
  `.claude/skills/`; `target` is the skill dir name in databricks-agent-skills
  (`skills/<target>/`); optional `related` is a cross-link to inject.
- `placeholders`: literal `{{...}}` → value substitutions applied to every
  synced file.

## Task

### 1. Locate repositories and check working trees

**In parallel:**
- **Source (this repo):** the current app-templates working directory.
- **Target (agent-skills):** `~/projects/databricks-agent-skills` or
  `~/databricks-agent-skills`. If not found, tell the user and stop.

Check both for uncommitted changes. If the agent-skills repo is dirty, or this
repo has uncommitted changes under `.claude/skills/`, ask the user — do **not**
stash or discard. Changes outside the sync scope (e.g. to this runbook) are fine.

### 2. Update to latest

Pull the latest default branch for each repo (`main` for both). If a pull fails,
ask the user.

### 3. Sync each skill

For every entry in `sync-config.yml` `skills`:

1. **Copy** `.claude/skills/<source>/` → `skills/<target>/` in the agent-skills
   repo (the whole dir: `SKILL.md`, `references/`, `examples/`, …).
2. **Render placeholders** in every text file (`.md`, `.markdown`, `.txt`,
   `.yaml`, `.yml`): replace each key in `placeholders` with its value. After
   rendering, **no `{{` may remain** (assert this in step 5).
3. **Rewrite the frontmatter `name:`** in the target `SKILL.md` to `<target>`.
4. **Rewrite sibling references:** anywhere the text says `` **<source>** skill ``
   or `` `<source>` skill `` (for any source name in the config), rewrite it to
   the sibling's `<target>` name. Leave bare prose names alone.
5. **Inject the cross-link:** if the entry has `related`, insert it as a callout
   immediately after the closing `---` of the frontmatter, on its own line:

   ```
   > **Related skill(s):** <related text>
   ```
6. **Write a provenance marker** `skills/<target>/.synced-from`:

   ```
   Generated by app-templates/sync-agent-skills (sync-agent-skills.md)
   Source of truth: github.com/databricks/app-templates (.claude/skills/<source>)
   Do not hand-edit — changes go to app-templates.
   ```

Process independent skills in parallel (e.g. with subagents) if there are many.

### 4. Remove stale mirrors

For any directory under `skills/` that has a `.synced-from` marker but whose name
is **not** a `target` in the config, delete it (its source skill was renamed or
removed). Never touch directories without a `.synced-from` marker — those are
hand-authored skills.

### 5. Assert clean render

Confirm no `{{` placeholder survives in any synced (`.synced-from`-marked)
directory. If any remain, stop and report which files.

### 6. New-skill prerequisites

For any **new** target (its dir did not previously exist in agent-skills), the
downstream `scripts/skills.py generate` (next step) auto-synthesizes
`agents/openai.yaml` + icons and adds the `manifest.json` entry. No
`SKILL_METADATA` / `plugin.json` entry is needed — synced skills are skipped by
that check.

For clean marketplace display names, framework/acronym targets (e.g.
`-langgraph`, `-openai`, `-api`) should have a `DISPLAY_NAME_OVERRIDES` entry in
the agent-skills `scripts/skills.py` (e.g. `Databricks Agent Tools (LangGraph)`),
otherwise the name auto-titlecases to `Langgraph` / `Openai` / `Api`. Add any
missing ones before generating.

### 7. Generate and validate the manifest

In the agent-skills repo:

```bash
python3 scripts/skills.py generate
python3 scripts/skills.py validate
```

`generate` syncs shared assets, synthesizes Codex metadata, and rebuilds
`manifest.json` in one step. If `validate` fails, fix the underlying issue before
proceeding — do not hand-edit `manifest.json`.

### 8. Summarize changes

Show the user a short bulleted summary of what was added / changed / removed.

### 9. Create a branch, commit, and push

In the agent-skills repo:

```bash
git checkout -b update/agent-template-skills-<YYYY-MM-DD>
```

Stage all changes (including deletions) and commit with a single-line message:

```
Update agent-template skills
```

Push the branch to a remote **you can write to**:
- If you have direct write to databricks-agent-skills, push to `origin`.
- Otherwise push to **your fork** (e.g. the `fork` remote /
  `<your-gh-username>/databricks-agent-skills`) — the contribution then comes
  from your fork, like any external PR.

### 10. Open the pull request (browser, no token)

Open the GitHub compare page in the user's browser with the PR pre-filled. Pick
the form that matches where you pushed:

```bash
# Pushed a branch on databricks-agent-skills directly:
open "https://github.com/databricks/databricks-agent-skills/compare/main...<branch>?expand=1&title=<url-encoded-title>&body=<url-encoded-body>"

# Pushed to your fork:
open "https://github.com/databricks/databricks-agent-skills/compare/main...<your-gh-username>:<branch>?expand=1&title=<url-encoded-title>&body=<url-encoded-body>"
```

A maintainer reviews and merges — this runbook only proposes.

**PR format:**

Title: `Update agent-template skills`

```markdown
## Summary
<1-3 sentences on what changed, written naturally>

### Changes
- **<skill>**: <one-line description>
- ...
```

**Safety (both repos are public):** app-templates is itself public, so there is
no internal content to scrub. Still: never include secrets, tokens, real
workspace IDs/URLs, or PATs in the branch or PR (the repo's secret-scan CI also
guards this), and keep the description about what the skills *do*.

## Error handling

- Source path in `sync-config.yml` doesn't exist → stop and help the user fix
  the config (offer to correct the `source`).
- A new source skill has no entry in `sync-config.yml` → it will simply not be
  synced. Add a `{ source, target }` entry (and `related` if it overlaps an
  existing skill) before re-running.
- Never force-push or overwrite uncommitted work in either repo.
